# Benchmarks

Task runners spend most of their wall-clock on *the work they launch*. To compare
the **runners themselves**, these scenarios run tasks that do almost nothing and
measure what's left: process startup, config parsing, and the cost of reaching
the child commands.

The same task definitions are generated for every tool (`tsr`, `npm`, `bun`,
`just`, go-task, `make`) ‚Äî see [`benches/`](https://github.com/Open-Tech-Foundation/tsr/tree/main/benches)
‚Äî and timed with [hyperfine](https://github.com/sharkdp/hyperfine). The numbers
below are loaded directly from the benchmark's JSON export.

## startup ‚Äî a single task

Spawns `true`. This is pure per-invocation overhead, and where the compiled
runners separate from the interpreter-based ones.

<BenchChart scenario={data.scenarios.startup} />

## in-task steps ‚Äî one launch, five commands

One task that runs five commands in sequence. Every runner launches **once** and
then spawns the child commands, so even `npm`/`bun` stay cheap here ‚Äî the startup
tax is paid a single time.

<BenchChart scenario={data.scenarios.steps5} />

## dependency graph ‚Äî five tasks

One task depending on five trivial tasks. `tsr`, `just`, go-task, and `make`
resolve the whole graph in a **single** launch. `npm`/`bun` have no dependency
graph, so ‚Äî as their users do ‚Äî the five tasks are chained with `&&`, paying the
runner's startup **five times over**.

<BenchChart scenario={data.scenarios.graph5} />

## dependency graph ‚Äî ten tasks

The same, scaled to ten dependencies. Overhead that is paid per-invocation grows
linearly: the graph-native runners barely move, while the chained `npm`/`bun`
costs roughly double from the five-task graph.

<BenchChart scenario={data.scenarios.graph10} />

## Takeaway

For a single task, `tsr` runs a metacharacter-free `run` string by spawning the
child directly (`execvp`-style) ‚Äî no language runtime, no wrapping shell ‚Äî so it
sits with the native runners (`make`, `just`) and well ahead of `npm run`. The
gap **compounds across a dependency graph**: because `tsr` resolves the entire
graph in one process, its cost stays flat while a chained `npm`/`bun` pays its
startup once per task. That is the case `tsr` is built for.

<Callout type="info" title="Not a build-performance claim">
  This measures runner overhead only. Content-hash and incremental caching are
  deliberately delegated to Turbo/Nx (see the [overview](/docs)) ‚Äî out of scope
  here.
</Callout>

<Callout type="warning" title="Your numbers will differ">
  Measured on one reference machine (Linux x86-64). Regenerate with

  to refresh this page.
</Callout>
