# Benchmarks

Task runners spend most of their wall-clock on *the work they launch*. To compare
the **runners themselves**, we run tasks that do almost nothing and measure what's
left: process startup, config parsing, and the cost of reaching the child command.

The same two tasks are defined for every tool (`tsr`, `npm`, `bun`, `just`,
go-task, and `make`) and timed with [hyperfine](https://github.com/sharkdp/hyperfine)
(`--shell=none`, 20 warmup runs). The numbers below are loaded directly from the
benchmark's JSON export — see [`benches/`](https://github.com/Open-Tech-Foundation/tsr/tree/main/benches).

## Takeaway

`tsr` runs a metacharacter-free `run` string by splitting it and spawning the
child directly (`execvp`-style) — no language runtime to boot and no wrapping
shell. It lands essentially tied with the native runners (`make`, `just`), a touch
ahead of `bun run`, and **~60–75× faster** than the interpreter-backed `npm run`
and go-task, which pay their startup on every invocation. This "no startup tax" is
the common-path win `tsr` is designed for.

<Callout type="info" title="Not a build-performance claim">
  This measures runner overhead only. Content-hash and incremental caching are
  deliberately delegated to Turbo/Nx (see the [overview](/docs)) — out of scope
  here.
</Callout>

<Callout type="warning" title="Your numbers will differ">
  Measured on one reference machine (Linux x86-64). Re-run <code>benches/run.sh</code>
  locally, then <code>node website/tools/sync-bench.mjs</code> to refresh this page.
</Callout>
